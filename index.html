Speaking at Arc of AI

I'll be presenting AEO (AI Engine Optimization): Writing Docs and Code for Machines at Arc of AI 2026 in Austin, Texas on April 15, 2026. The talk makes the case that SEO is dead for developers — the new workflow for building software has shifted from the Google search bar to the IDE prompt box. When a developer asks an AI agent like Claude, Cursor, or a custom MCP server to implement a library or integrate an API, they're no longer the primary consumer of your documentation. The LLM is.

If your code, documentation, and reference architectures aren't optimized for machine ingestion, the AI will hallucinate the implementation — and the developer will blame your product. The session covers the emerging standards for addressing this, including the llms.txt proposal and Andrew Ng's Context-Hub, and dives into practical techniques: trimming "marketing fluff" to preserve the context budget for actual logic, and structuring Python and JavaScript repositories so AI agents can parse abstract syntax trees without ambiguity.

Hammerspace

From January to the end of March 2026, I worked as a contract developer at Hammerspace, a data orchestration and global file system company, referred by my friend Mike Bloom, their Director of AI Architecture, with whom I worked at OpenCola.

The objective: optimize a Python-based MCP (Model Context Protocol) server to handle industrial-scale file ingest events — a system that was going to be the backbone of a live demo for NVIDIA at GTC 2026.

The Problem

The original system followed a reactive waterfall pattern: for every file event detected, it spawned multiple subprocesses — to hash the file, query a database, and log the result. At small scale, this worked fine. But when subjected to the kind of "event storm" that happens when thousands of files are dropped simultaneously, the server attempted to spawn 2,000+ Linux processes at once, creating a fork bomb that consumed all CPU cycles for kernel-level context switching, caused database connection timeouts, and eventually hung the entire system.

The performance at the start of my contract: 60 files per second.
The target: 1,000+ files per second.

The Solutions

Fix 1: Proactive Snapshotting (eliminating the fork bomb)

I refactored the discovery client from a reactive to a proactive architecture. Instead of making a subprocess call per file to check whether it was already known to the system, the client now pulls a full snapshot of known file hashes into memory every 10 seconds and performs all existence checks as in-memory lookups.

Result: per-file existence checks went from ~200ms (CLI subprocess overhead) to under 0.01ms (RAM lookup) — a 20,000x speedup for that operation — and the fork bomb was eliminated entirely.

Fix 2: Sampled Hashing (eliminating the I/O bottleneck)

Hashing multi-gigabyte files — like the model weights and datasets common in AI workflows — in their entirety created an I/O bottleneck. I implemented a sampled fingerprinting approach: instead of reading the whole file, the hasher reads 128KB chunks from the head, middle, and tail of each file. To maintain integrity, the file's total size is mixed into the final hash, ensuring that two files with identical sampled chunks but different total sizes produce different fingerprints.

Result: hashing became effectively constant-time regardless of file size.

Results

Metric Before After
Throughput 60 files/sec 420 files/sec (targeting 1,000+)
CPU efficiency 2 files/sec per 1% CPU 10.85 files/sec per 1% CPU
Existence check latency ~200ms <0.01ms
 

The work was presented at NVIDIA's GTC 2026 conference as part of a live AI data platform demo, and was featured in Hammerspace's NVIDIA AI Data Platform integration work.

Technologies

Python, MCP (Model Context Protocol), SHA-256 sampled hashing, MySQL, Linux process management, distributed file systems, AI/ML infrastructure

Kansas City Developer Conference 2025: The Best, Most Fun Python Platform You’ve Never Heard Of.

I gave a talk at the 2025 Kansas City Developer Conference titled The Best, Most Fun Python Platform You’ve Never Heard Of. It was a programmer’s introduction to the powerful, fun, and all-too-often-ignored Ren’Py visual novel engine.

Here’s the abstract for the talk:

Python’s occupied the number one spot on the TIOBE Programming Community Index for the past couple of years, and it’s the preferred programming language in for AI and data science. Perhaps you’ve been thinking about learning it, but the thought of having to do another set of “Hello World” style exercises is filling you with dread. Is there a more fun way to get up to speed with Python?

Yes, there is, and it’s called Ren’Py. It’s billed as a visual novel engine and often used for writing dating simulation games, but it’s so much more than that. It’s a platform that lets you code in Python (and more) and deploy to desktop, web, and even mobile platforms, and with a fraction of the effort required by React, Vue, or Angular. It’s a fun framework that’s been used to produce games you can find on Steam, but it’s got applications well beyond amusement.

In this session, we’ll look not just at the basics of Ren’Py development, but the building of a dating game based on KFC’s official game, “I Love You Colonel Sanders,” a simple turn-based combat game starring Florida Man, and building mobile apps in a way that’s less frustrating than usual.

My own tech blog, Global Nerdy

My tech blog, Global Nerdy, has had over 10 million pageviews since I started it in 2006. It’s a blog about technology, programming, and the occasional bit of nerd culture. I’ve posted almost 5,000 articles there.

Since 2017, I’ve been publishing a weekly list of Tampa Bay tech events on Global Nerdy. I originally built the list manually, when it the list was small and it was a job search tactic. When that job search ended, I’d planned to wind down the list, only to be asked “Hey, where’s the list?”

It turned out that the list was useful to the local tech community, and I was happy to keep it going. However, as the list grew, it became more and more of a chore to maintain. I’d have to check Meetup.com for new events, copy and paste them into the list, and then publish it.

To solve that problem, I automated the process of gathering and publishing the list using a Python script that scrapes Meetup.com for local tech, entrepreneur, and “nerd” events.I’ve published the source code for that script on my GitHub account.
